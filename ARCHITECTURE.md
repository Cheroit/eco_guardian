# EcoGuardian: сценарии, архитектура и структура данных (черновик)

## 1) Цель прототипа
EcoGuardian — учебный сервис, который:
1) получает (или синтетически генерирует) экологические показатели для города;
2) рассчитывает интегральную оценку риска по правилам/движку;
3) использует LLM, чтобы превращать численные данные в понятные объяснения и рекомендации, а также отвечать на вопросы пользователя.

Важно: LLM отвечает за **интерпретацию и язык**, а “числа” (например, пороговые оценки/веса) — за **детерминированный риск-движок**. Это снижает риск “галлюцинаций” и делает результат проверяемым.

---

## 2) Сценарии использования (LLM-ориентированные)

### Сценарий A. “Текущая обстановка” (дашборд)
**Пользователь** выбирает город и нажимает “Анализировать”.
**Сервис**:
1. строит `CityEnvironmentSnapshot` (показатели + timestamp);
2. запускает `RiskEngine` → `RiskAssessment`;
3. формирует “объяснение” и список “действий” через LLM (опираясь на snapshot и оценку риска);
4. возвращает в UI данные для визуализации (risk_level, score, summary, рекомендации).

**Что важно для задания:** объяснение и рекомендации в прототипе могут быть синтетическими, но архитектурно они должны генерироваться LLM на основе структурированных данных.

### Сценарий B. “Вопрос-ответ” (Q&A о безопасности)
**Пользователь** вводит вопрос: “Насколько безопасно сегодня заниматься спортом на улице?”
**Сервис**:
1. (опционально) подтягивает текущий snapshot для города (или использует snapshot из последнего вызова);
2. формирует prompt, где LLM получает:
   - численные показатели,
   - `risk_level`, `risk_overall_score`,
   - краткий `risk_type_breakdown`,
   - вопрос пользователя;
3. LLM отвечает в фиксированном формате (см. контракт в разделе 5): короткий ответ + обоснование + рекомендации с учетом контекста (спорт/время/интенсивность).

### Сценарий C. “Пояснить факторы” (deep-dive)
**Пользователь** просит: “Почему риск высокий?” или “Что сильнее всего повлияло?”
**Сервис**:
1. берёт `risk_type_breakdown` и исходные метрики из snapshot;
2. LLM формирует причинно-следственное объяснение: “значения → какой компонент риска → что это значит для поведения”.

Этот сценарий полезен для демонстрации “structure + reasoning” от LLM.

## 3) Первичная архитектура (модули и взаимодействия)

Ниже — архитектура, в которую текущий `backend/main.py` можно “развернуть” в отдельные файлы.

### 3.1 Компоненты
1. **API слой (FastAPI)**  
   Принимает HTTP запросы, валидирует входы (Pydantic), отдает ответы.

2. **Environment Provider / Data Layer**
   - в прототипе: `MockEnvironmentProvider` (синтетика);
   - в будущем: реальные провайдеры данных (погода, качество воздуха и т.п.).
   - обеспечивает единый интерфейс: `get_snapshot(city, timestamp) -> CityEnvironmentSnapshot`.

3. **Risk Engine**
   Детерминированный расчет:
   - `RiskAssessment` по `EnvironmentalReading`.
   - фиксированные веса/пороговые значения.

4. **LLM Orchestrator**
   - строит prompt и context,
   - вызывает LLM API,
   - валидирует output (лучше — JSON-структура),
   - возвращает “готовый” результат API-слою.

5. **Models / Schemas**
   Pydantic-объекты (и/или JSON Schema) для:
   - snapshot,
   - риска,
   - контракта LLM.

### 3.2 Поток для сценариев (общая схема)
1) `API` → `EnvironmentProvider` → `RiskEngine` → `LLMOrchestrator` → `API response`  
Где LLMOrchestrator получает **все необходимые входные данные строго в структуре**, а не только “сырой текст”.

## 4) Структура данных (контракты и примеры)

### 4.1 Snapshot (вся “снимочная” информация по городу)
`CityEnvironmentSnapshot`
```json
{
  "city": "Moscow",
  "timestamp": "2026-03-18T20:33:03Z",
  "reading": {
    "temperature": 5.2,
    "humidity": 42.1,
    "wind_speed": 3.4,
    "precipitation": 0.0,
    "air_quality_index": 84.0,
    "pm25": 18.7,
    "pm10": 33.2,
    "water_level": 1.2,
    "fire_danger_index": 55.3
  }
}
```

### 4.2 Оценка риска (численная, “проверяемая”)
`RiskAssessment`
```json
{
  "overall_risk_score": 63.4,
  "risk_level": "high",
  "risk_type_breakdown": {
    "fire": 51.0,
    "flood": 20.0,
    "air_pollution": 72.0
  },
  "engine_notes": {
    "weights": { "fire": 0.4, "flood": 0.25, "air_pollution": 0.35 }
  }
}
```

### 4.3 Контекст для LLM (то, что реально отправляется в prompt)
`LLMContext`
```json
{
  "scenario": "dashboard" | "qa" | "explain_factors",
  "city": "Moscow",
  "snapshot": { "...": "CityEnvironmentSnapshot" },
  "risk": { "...": "RiskAssessment" },
  "user_input": {
    "question": "string",
    "activity": "optional string",
    "time_of_day": "optional string"
  }
}
```

### 4.4 Схема ответа API (пример для “текущей обстановки”)
`CurrentEnvironmentResponse`
```json
{
  "city": "Moscow",
  "reading": { "...": "EnvironmentalReading" },
  "risk": {
    "overall_risk_score": 63.4,
    "risk_level": "high",
    "summary": "string (LLM)",
    "recommendations": ["string (LLM)"],
    "risk_type_breakdown": { "...": "числа" }
  }
}
```

## 5) Контракт LLM-ответа (чтобы результат был структурированным)

Чтобы прототип был “инженерным”, LLM лучше просить вернуть JSON.

### 5.1 Контракт для summary+recommendations
`LLMGuidance`
```json
{
  "summary": "Короткое объяснение в 1-3 предложения.",
  "recommendations": [
    { "action": "Избегать интенсивной нагрузки на улице", "priority": "high", "reason": "PM2.5/air_pollution высокий" },
    { "action": "Проверить сводку о воде/ливнях", "priority": "medium", "reason": "flood компонент риска поднят" }
  ],
  "safety_disclaimer": "учебный прототип/не для критически важных решений"
}
```

### 5.2 Контракт для Q&A
`LLMQAAnswer`
```json
{
  "answer": "Короткий ответ пользователю.",
  "reasoning": "Почему так (с опорой на метрики и risk_type_breakdown).",
  "recommended_actions": [
    { "action": "string", "priority": "low|medium|high", "evidence": ["pm25", "aqi", "fire_danger_index"] }
  ]
}
```

## 6) Как это ложится на текущий код (bridge к рефакторингу)
Сейчас в `backend/main.py` уже есть:
- `EnvironmentalReading`,
- `RiskAssessment`,
- endpoint `GET /api/environment/current`,
- endpoint `POST /api/qa`.

Следующий шаг, чтобы “доказать LLM-подход” прямо архитектурой:
1. вынести `RiskEngine` в отдельную функцию/модуль;
2. добавить `LLMOrchestrator`, который:
   - получает snapshot + risk,
   - генерирует `summary`/`recommendations`,
   - генерирует ответ в Q&A по фиксированному контракту JSON.
